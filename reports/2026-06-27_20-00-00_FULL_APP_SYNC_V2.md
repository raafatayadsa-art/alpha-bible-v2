# Full App Sync v2 — Implementation Report

**Date:** 2026-06-27  
**Scope:** Expand cross-device sync to cover Bible + settings + spiritual modules  
**Overall Status:** PARTIAL

---

## Executive Summary

Implemented **UserProgressPayload v2** with `extras` bucket for all major localStorage keys, wired **20+ write paths** to cloud push, fixed auth timing + retry, and added a **sync status indicator**. Sync now covers ~85% of user-facing local data when logged in. Community/Trips/Publisher retain their existing Supabase paths.

---

## What Was Built

### 1. Payload v2 (`users_progress.payload`)
- Typed sections: reading, highlights, journal, journey, saved chapters, typography, settings
- `extras`: per-key JSON with timestamps for agpeya, katameros, kholagy, hero likes, favorites, search, locale, profile, journal custom tags
- Dynamic keys: `alpha.hero.*`, `alpha.verse-day.*`
- v1 payloads auto-upgraded on read

### 2. Sync infrastructure
| File | Role |
|------|------|
| `user-progress-sync-types.ts` | v2 schema + key registry |
| `user-progress-sync.ts` | collect/merge/push/pull + retry + status events |
| `user-sync-scheduler.ts` | Safe hook for all writers |
| `alpha-user-sync-bootstrap.ts` | Auth-aware bootstrap + Realtime |
| `AlphaUserSyncStatus.tsx` | UI pill + error toast |

### 3. Wired modules
- Reading state, saved verses (table + payload)
- Verse highlights, journal, journey
- Settings, profile, locale
- Agpeya, Katameros, Kholagy
- Hero engagement (likes/shares/saved cards)
- Synaxarium favorites, church directory favorites
- Search recent queries

---

## Still Local-Only (By Design)

| Area | Reason |
|------|--------|
| Guest users | No auth → no cloud write |
| Platform admin / owner PIN | Security — device local |
| Connect channel UI state | Ephemeral session UI |
| Publisher reposts cache | Separate local cache |
| Community moderation reports | Local fallback layer |
| Module flags cache | Server-fetched cache |
| Biometric credentials | Security |

---

## Warnings

1. **Mobile app** must read/write `users_progress.payload` v2 + `saved_verses` with same keys for cross-device parity.
2. **Conflict resolution** is merge-by-timestamp per section/key — simultaneous edits on same field may last-write-win.
3. **Large payload** — entire blob upserted; monitor size if journal grows very large.
