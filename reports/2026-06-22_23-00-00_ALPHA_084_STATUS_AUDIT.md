# ALPHA-084 — Trip Organizer Role & Approval Workflow

**Date:** 2026-06-22  
**Status in spec:** Approved Idea  
**Implementation:** **v1 shipped (client-side)** — production hardening pending  
**Category:** Church Management / Reservations / Permissions

---

## Executive Summary

ALPHA-084 is **largely implemented in code** as an isolated `trip-organizer` module wired into `/church`, PostBuilder, trip reservations, and ALPHA-083 dual channels. The full workflow works: grant **منظم رحلات** → create trip → **Pending Approval** → priest/servant reviews → approve publishes post + opens bookings + provisions Trip + Organizer channels with audit fields.

**Main gap:** grants, reviewers, and channel assignments live in **localStorage** (per device), not Supabase — so multi-user production churches need a DB migration before relying on this in the field.

## Findings — Spec vs implementation

### 1. New role: منظم رحلات ✅ (v1)

| Spec | Implementation |
|------|----------------|
| Role separate from priest / servant / deacon / admin | `isTripOrganizerOnly()` — grant without `canManageChurchPosts` |
| Priest or admin can grant | `canGrantTripOrganizerRole()` — owner, priest, **servant** |
| Grant to servant, deacon, member, activity lead | UI: manual `user-id` + name in `TripOrganizerGrantSheet` |
| Scope: trips, conferences, monastery visits, reservations, participants | `canCreateTripContent()` — **trip post type only** in v1 |
| **NOT** church admin, servants admin, members, church data, church-wide notifications | Enforced: organizers blocked from post management surfaces unless also priest/servant |

## Architecture (current)

```
Priest grants role (localStorage)
        ↓
Organizer → PostBuilder → submitTripPost()
        ↓
pending (hidden from feed, no channels, no booking)
        ↓
Reviewer → TripApprovalSheet
        ↓
approve → patchChurchPostDetails + provisionTripChannels()
        ↓
Public feed + reservations + trip_public + trip_organizer
```

---

## Warnings

1. **localStorage grants** — another admin/device cannot see grants; organizers lose access on new browser.
2. **No server-side enforcement** — RLS does not validate trip-organizer role; savvy client could bypass (needs RPC).
3. **All servants can review** — may be broader than spec (“خادم مخول بالمراجعة” only).
4. **Servants can grant role** — spec says priest or admin; code includes `servant`.
5. **Conferences / monastery visits** — spec lists them; v1 uses single `trip` post type.
6. **Resubmit after `changes_requested`** — organizer can edit post but no dedicated “re-submit for approval” UX.
7. **Alpha Connect dependency** — channels provision locally in registry; not synced to Supabase Connect backend if disconnected.

## Recommendations — Production hardening (priority order)

### P0 — Before field use

1. **Migration:** `trip_organizer_grants` table (`church_id`, `user_id`, `scope`, `linked_post_id`, `granted_by`, `expires_at`) + RLS.
2. **RPC:** `grant_trip_organizer`, `revoke_trip_organizer`, `can_create_trip(church_id)`.
3. **Server check** in `createChurchPost` / `patchChurchPostDetails` for approval transitions.

### P1 — UX & spec alignment

4. Member picker from `church_memberships` instead of raw user-id.
5. Reviewer picker UI (`grantTripReviewer` already exists).
6. In-app notification to priest on `submitTripPost` (church notifications table or existing bell).
7. Resubmit flow when `changes_requested`.

### P2 — Scope expansion

8. Post subtypes: `conference`, `monastery_visit` or tags on trip posts.
9. Restrict `canGrantTripOrganizerRole` to priest + owner only (config flag for servants).
10. Trip organizer dashboard (pending / approved / my trips).
